## CONSTRAINTS
- HARD: API is stateless — no fan state tracking. Fans ignore redundant commands. Client handles idempotency.
- HARD: No authentication on API — Tailscale VPN is the auth layer
- HARD: No external server — ESP32 IS the server. Self-contained.
- HARD: LittleFS.begin(false) — never auto-format. POST /factory-reset is the intentional wipe path.
- SOFT: PlatformIO + Arduino framework (not raw ESP-IDF). MicroLink may require ESP-IDF — evaluate when implementing Tailscale.
- SOFT: API uses POST + action verbs in paths (user preference, not REST orthodoxy)

## DECISIONS MADE
- **ID-based commands** → cleaner URLs, no encoding issues. GET /fans returns id+name+capabilities for client to build UI.
- **Learn via API** (POST /fans/{id}/learn/{command}) not serial monitor → no toolchain needed after flash. User presses remote, code saved automatically.
- **FreeRTOS task for rfLearn** → POST /learn returns 202 immediately; poll GET /fans/{id}/learn/status. Prevents blocking async_tcp task for 30s.
- **Chunked body accumulator** (g_bodyBufs map) → correct multi-chunk JSON parsing, not just last chunk.
- **reverse as separate on/off commands** (reverse_on/reverse_off) → user needs to un-reverse fans, not just toggle. Same pattern as lights.
- **MicroLink deferred** → layer in after core API confirmed working. If too painful, API accessible on LAN without Tailscale.
- **Partition table custom** → 16MB flash, 9.9MB LittleFS for config + room for OTA.
- **POST /factory-reset** → explicit wipe endpoint instead of silent format-on-fail.
- **Web UI: single HTML file in LittleFS** → `data/index.html`, no build step, vanilla JS, served via `GET /`. No CDN — fully offline capable.
- **WiFi change via API** → POST /wifi/update saves new creds to NVS + reboots. No factory reset needed to change networks.

## WHAT FAILED / REJECTED
- REST PUT for update → user rejected. Use POST /fans/{id}/update with action verbs.
- Serial-based RF sniffing → replaced with learn-via-API (POST /learn endpoint). Better UX.
- Blocking rfLearn in onRequest handler → fixed. Was freezing web server for 30s.
- LittleFS.begin(true) → removed. Was silently wiping all learned codes on any mount hiccup.
- String literal concatenation with constexpr const char* → compile error in wifi_manager.cpp, fixed to Serial.print() calls.

## REASONING PATTERNS
- SITUATION: API design choices → WRONG INSTINCT: follow REST conventions (PUT, resource-centric) → CORRECT FRAMING: user prefers readable verb-in-path APIs (POST /fans/add, not POST /fans). Apply this to any new endpoints.
- SITUATION: AsyncWebServer + long-running operations → WRONG INSTINCT: block in onRequest → CORRECT FRAMING: offload to FreeRTOS task, return immediately, add poll endpoint. async_tcp task must never block.
- SITUATION: validating capabilities per fan → CORRECT FRAMING: lights and reverse are both capability flags. Any new capability follows same pattern: bool field + command validation in validateCommandForFan + paired on/off endpoints.

## PLANNED WORK

### WiFi Change API
`POST /wifi/update` — change WiFi without factory reset.
- Body: `{"ssid":"...","password":"..."}`
- Validate: ssid non-empty ≤32, password 8-63 or empty
- Calls `nvsSaveCreds()` (already in wifi_manager.cpp, needs exposing as public)
- Returns `{"ok":true}` then reboots after 1.5s
- Files: `wifi_manager.h` (expose saveCreds), `wifi_manager.cpp` (make public), `api.cpp` (add route)

### Web UI
Single-page app served from LittleFS at `GET /`. File: `data/index.html` (~400-600 lines, vanilla JS, no CDN).
- Upload via `pio run -t uploadfs`
- Tabs: Fans (control) | Setup (add/edit/learn) | WiFi (change creds) | System (status + factory reset)
- Learn flow: press Learn → 30s countdown → auto-polls GET /fans/{id}/learn/status → shows result
- Mobile-first, card layout per fan, green=learned / grey=not learned
- Files: `data/index.html` (new), `api.cpp` (add `GET /` serving index.html from LittleFS)
- Serve with: `req->send(LittleFS, "/index.html", "text/html")`
